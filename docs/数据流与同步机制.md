# BookmarkHub 数据流与同步机制

**版本**: 0.7 | **更新日期**: 2026-05-11

## 1. 同步模式概述

BookmarkHub 支持三种同步模式，可灵活配置以满足不同使用场景。

### 1.1 同步模式对比

```mermaid
graph TB
    subgraph 同步模式
        Interval[定时同步]
        Event[事件触发同步]
        Hybrid[混合同步]
    end
    
    subgraph 定时同步触发
        Alarm[Alarm API 定时触发]
        Interval --> Alarm
    end
    
    subgraph 事件触发源
        Startup[浏览器启动]
        BookmarkChange[书签变更事件]
        UserAction[用户手动操作]
        Event --> Startup
        Event --> BookmarkChange
        Event --> UserAction
    end
    
    subgraph 混合模式
        Both[同时启用定时+事件]
        Hybrid --> Both
        Both --> Alarm
        Both --> BookmarkChange
    end
```

| 模式 | 配置项 | 触发条件 | 适用场景 |
|------|--------|----------|----------|
| **定时同步** | `enableIntervalSync` | Alarm 定时器触发 | 固定周期自动同步 |
| **事件触发** | `enableEventSync` | 书签变更/浏览器启动 | 实时响应变更 |
| **混合模式** | 同时启用 | 定时 + 事件双重触发 | 最大同步保障 |

## 2. 同步流程详解

### 2.1 完整同步流程图

```mermaid
sequenceDiagram
    participant User as 用户操作
    participant Popup as Popup 界面
    participant BG as Background Worker
    participant Sync as Sync Engine
    participant Merge as Merge Engine
    participant Detect as Change Detection
    participant Cache as Local Cache
    participant Remote as 远程服务
    
    User->>Popup: 点击同步按钮
    Popup->>BG: 发送 sync 消息
    BG->>BG: 检查 isSyncing 锁
    BG->>Sync: performSync()
    
    Sync->>Sync: 设置同步锁
    Sync->>Sync: 设置事件抑制标志
    Sync->>BG: 获取设置 Setting.build()
    Sync->>BG: 获取本地书签 getBookmarks()
    Sync->>Remote: 获取远程数据 fetchRemoteData()
    
    Remote-->>Sync: 返回远程书签数据
    Sync->>Sync: 标准化书签 ID
    Sync->>Cache: 获取本地缓存 getLocalCache()
    Cache-->>Sync: 返回基准点 baseline
    
    Sync->>Detect: 检测本地变更 detectChanges(baseline, local)
    Sync->>Detect: 检测远程变更 detectChanges(baseline, remote)
    Detect-->>Sync: 返回变更检测结果
    
    Sync->>Merge: 执行三向合并 threeWayMerge()
    Merge->>Merge: 处理墓碑 tombstones
    Merge->>Merge: 检测冲突 findConflicts()
    Merge->>Merge: 解决冲突 resolveConflicts()
    Merge->>Merge: 应用变更 applyChangesToBaseline()
    Merge-->>Sync: 返回合并结果
    
    Sync->>Remote: 上传合并数据 uploadBookmarks()
    Sync->>Cache: 更新本地缓存 saveLocalCache()
    Sync->>Sync: 保存同步状态
    Sync->>Sync: 释放同步锁
    Sync->>Popup: 发送 refreshCounts 消息
    Sync-->>BG: 返回 SyncResult
    BG-->>Popup: 返回结果
    Popup->>User: 显示同步结果
```

### 2.2 同步流程步骤详解

| 步骤 | 操作 | 说明 |
|------|------|------|
| **1. 锁检查** | `isSyncing` | 防止并发同步，已有同步进行中则跳过 |
| **2. 设置锁** | `isSyncing = true` | 设置同步锁和事件抑制标志 |
| **3. 获取配置** | `Setting.build()` | 从存储读取用户配置 |
| **4. 本地书签** | `getBookmarks()` | 获取浏览器完整书签树 |
| **5. 远程数据** | `fetchRemoteData()` | 从 Gist/WebDAV 获取云端数据 |
| **6. ID 标准化** | `normalizeBookmarkIds()` | 确保本地远程使用相同稳定 ID |
| **7. 基准点获取** | `getLocalCache()` | 获取上次同步状态作为基准 |
| **8. 变更检测** | `detectChanges()` | 检测本地和远程相对基准的变更 |
| **9. 三向合并** | `threeWayMerge()` | 合并两边变更，处理冲突 |
| **10. 上传数据** | `uploadBookmarks()` | 上传合并后的数据到云端 |
| **11. 缓存更新** | `saveLocalCache()` | 更新本地缓存为新基准点 |
| **12. 释放锁** | `isSyncing = false` | 释放同步锁，恢复事件监听 |

## 3. 三向合并算法

### 3.1 合并原理

三向合并是分布式同步的核心算法，原理如下：

```mermaid
graph TB
    subgraph 时间线
        T0[T0: 上次同步<br/>基准点 baseline]
        T1[T1: 本地当前<br/>local]
        T2[T2: 远程当前<br/>remote]
    end
    
    subgraph 变更检测
        D1[本地变更:<br/>baseline → local]
        D2[远程变更:<br/>baseline → remote]
    end
    
    T0 --> D1
    T0 --> D2
    D1 --> T1
    D2 --> T2
    
    subgraph 合并结果
        M1[合并本地变更]
        M2[合并远程变更]
        Merge[最终合并结果]
    end
    
    T0 --> M1
    T0 --> M2
    M1 --> Merge
    M2 --> Merge
```

**核心思想**:
- 以 **baseline（上次同步状态）** 作为共同参考点
- 检测 **本地相对 baseline 的变更**
- 检测 **远程相对 baseline 的变更**
- **合并两边变更**，相同变更直接应用，冲突变更按策略处理

### 3.2 变更类型

```mermaid
graph LR
    subgraph 变更类型
        Created[新增<br/>created]
        Modified[修改<br/>modified]
        Deleted[删除<br/>deleted]
        Moved[移动<br/>moved]
    end
    
    subgraph 检测逻辑
        C1[baseline 无, local 有]
        M1[baseline 有, local 属性变]
        D1[baseline 有, local 无]
        MV1[parentId 或 index 变]
    end
    
    C1 --> Created
    M1 --> Modified
    D1 --> Deleted
    MV1 --> Moved
```

| 变更类型 | 检测条件 | 合并处理 |
|----------|----------|----------|
| **新增** | baseline 无此 ID，当前有 | 合并到结果树 |
| **修改** | ID 相同，title/url 属性变化 | 按时间戳决定版本 |
| **删除** | baseline 有此 ID，当前无 | 应用删除，创建墓碑 |
| **移动** | parentId 或 index 变化 | 按时间戳决定位置 |

### 3.3 冲突处理

```mermaid
graph TB
    subgraph 冲突场景
        BothMod[双方同时修改<br/>同一书签]
        LocalDel[本地删除<br/>远程修改]
        RemoteDel[远程删除<br/>本地修改]
    end
    
    subgraph 解决策略
        Auto[自动模式<br/>时间戳优先]
        Prompt[提醒模式<br/>用户选择]
    end
    
    BothMod --> Auto
    BothMod --> Prompt
    LocalDel --> Auto
    LocalDel --> Prompt
    RemoteDel --> Auto
    RemoteDel --> Prompt
    
    Auto --> Winner[修改时间晚者获胜]
    Prompt --> UserChoice[等待用户决定]
```

**自动模式冲突解决规则**:
- **修改冲突**: 比较 `dateGroupModified` 或 `dateAdded`，时间晚者获胜
- **删除优先**: 删除操作优先级高于修改（防止数据泄露）
- **新增不冲突**: 双方新增不同 ID 的书签，直接合并

## 4. 墓碑机制

### 4.1 墓碑原理

墓碑（Tombstone）是分布式同步中解决删除传播问题的核心机制。

```mermaid
sequenceDiagram
    participant DevA as 设备 A
    participant Cloud as 云端
    participant DevB as 设备 B
    
    Note over DevA,DevB: 初始状态：双方都有书签 X
    
    DevA->>DevA: 用户删除书签 X
    DevA->>DevA: 创建墓碑记录
    DevA->>Cloud: 上传（含墓碑）
    Cloud-->>DevA: 上传成功
    
    DevB->>Cloud: 同步（获取墓碑）
    Cloud-->>DevB: 返回数据（含墓碑）
    
    DevB->>DevB: 发现书签 X 在墓碑中
    DevB->>DevB: 删除本地书签 X
    DevB->>DevB: 跳过书签 X 的创建（防复活）
    
    Note over DevA,DevB: 最终状态：双方都无书签 X
```

### 4.2 墓碑数据结构

```typescript
interface Tombstone {
    /** 被删除书签的稳定 ID */
    id: string;
    /** 删除时间戳（毫秒） */
    deletedAt: number;
    /** 删除设备标识 */
    deletedBy: string;  // e.g. "Chrome/Windows"
}
```

### 4.3 墓碑生命周期

```mermaid
graph LR
    subgraph 墓碑创建
        Delete[书签删除事件]
        Create[创建墓碑记录]
        Local[存入本地缓存]
    end
    
    subgraph 墓碑传播
        Upload[上传到云端]
        Download[其他设备下载]
        Apply[应用删除操作]
    end
    
    subgraph 墓碑清理
        TTL[保留 30 天]
        Cleanup[过期自动清理]
    end
    
    Delete --> Create --> Local --> Upload --> Download --> Apply
    Create --> TTL --> Cleanup
```

**墓碑管理规则**:
- **保留周期**: 30 天（可在 `cleanExpiredTombstones` 中调整）
- **合并策略**: 相同 ID 的墓碑保留最新时间戳
- **防复活**: 变更检测时过滤掉墓碑中的书签创建

## 5. 事件抑制机制

### 5.1 事件抑制原理

同步操作会批量修改书签，如果不抑制事件，会导致递归同步触发。

```mermaid
sequenceDiagram
    participant Sync as 同步引擎
    participant Event as 事件监听器
    participant BM as 书签 API
    
    Note over Sync: 开始同步
    
    Sync->>Sync: isSuppressingEvents = true
    
    Sync->>BM: 批量创建书签
    BM-->>Event: 触发 onCreated 事件
    Event->>Event: 检查 isSuppressingEvents
    Event-->>Event: 跳过触发同步
    
    Sync->>BM: 批量删除书签
    BM-->>Event: 触发 onRemoved 事件
    Event->>Event: 检查 isSuppressingEvents
    Event-->>Event: 跳过触发同步
    
    Note over Sync: 同步完成
    Sync->>Sync: isSuppressingEvents = false
    
    BM-->>Event: 后续事件正常触发
    Event->>Sync: 触发同步（正常流程）
```

### 5.2 事件监听器结构

```typescript
// sync.ts 事件监听器定义
const syncListeners = {
    onStartup: () => {
        if (!isSuppressingEvents) {
            syncDebouncer.triggerSync();
        }
    },
    onCreated: (id, bookmark) => {
        if (!isSuppressingEvents) {
            syncDebouncer.triggerSync();
            executeCallbacks('onCreated', id, bookmark);
        }
    },
    // ... 其他监听器类似
};
```

## 6. MV3 Service Worker 兼容

### 6.1 MV3 挑战

Chrome MV3 中 Service Worker 会休眠，导致：
- 定时器失效
- 内存状态丢失
- 事件监听器注销

### 6.2 解决方案

```mermaid
graph TB
    subgraph MV3 兼容方案
        Alarm[Alarm API<br/>替代定时器]
        Storage[Storage API<br/>持久化状态]
        Wake[Service Worker<br/>唤醒恢复]
    end
    
    subgraph 状态持久化
        SyncState[同步锁状态]
        Cache[本地缓存]
        Timestamp[时间戳校验]
    end
    
    Alarm --> SyncState
    Storage --> Cache
    Storage --> Timestamp
    
    Wake --> SyncState
    Wake --> Cache
```

| 问题 | 解决方案 | 实现文件 |
|------|----------|----------|
| 定时同步失效 | Alarm API | `sync.ts:305` |
| 同步锁丢失 | Storage 持久化 | `sync.ts:saveSyncState()` |
| 状态恢复 | 时间戳校验（5分钟过期） | `sync.ts:restoreSyncState()` |

### 6.3 Alarm 配置

```typescript
// MV3 Alarm 监听器
browser.alarms.onAlarm.addListener((alarm) => {
    if (alarm.name === MV3_CONFIG.SYNC_ALARM_NAME) {
        performSync().catch(err => logger.error('alarm sync failed', err));
    }
});

// 创建 Alarm（最小间隔 1 分钟）
browser.alarms.create(MV3_CONFIG.SYNC_ALARM_NAME, {
    periodInMinutes: Math.max(setting.syncInterval / 60, 1)
});
```

## 7. 防抖机制

### 7.1 防抖原理

书签变更事件可能短时间内触发多次，防抖确保只执行一次同步。

```mermaid
sequenceDiagram
    participant Events as 书签事件
    participant Debounce as 防抖器
    participant Sync as 同步引擎
    
    Events->>Debounce: 事件 1 触发
    Debounce->>Debounce: 开始计时（等待 5 秒）
    
    Events->>Debounce: 事件 2 触发（2秒后）
    Debounce->>Debounce: 重置计时
    
    Events->>Debounce: 事件 3 触发（3秒后）
    Debounce->>Debounce: 重置计时
    
    Note over Debounce: 5 秒无新事件
    Debounce->>Sync: 执行同步
    
    Note over Debounce: 最大等待 30 秒
    Debounce->>Sync: 强制执行同步
```

### 7.2 防抖配置

```typescript
// 默认配置
const DEBOUNCE_CONFIG = {
    debounceTime: 5000,    // 5 秒防抖等待
    maxWaitTime: 30000,    // 30 秒最大等待
};
```

## 9. 状态管理

### 9.1 同步状态定义

```typescript
interface SyncState {
    /** 是否正在同步 */
    isSyncing: boolean;
    /** 是否抑制事件 */
    isSuppressingEvents: boolean;
    /** 状态时间戳 */
    timestamp: number;
}
```

### 9.2 状态流转

```mermaid
stateDiagram-v2
    [*] --> Idle: 初始状态
    
    Idle --> Syncing: 开始同步
    Syncing --> Idle: 同步成功
    Syncing --> Idle: 同步失败
    
    Idle --> Suppressing: 设置抑制标志
    Suppressing --> Idle: 清除抑制标志
    
    Syncing --> Suppressing: 同步中
    Suppressing --> Syncing: 保持抑制
```

## 10. 错误处理流程

### 10.1 错误分类

```mermaid
graph TB
    subgraph 错误类型
        Auth[认证错误]
        Network[网络错误]
        Data[数据错误]
        Validation[验证错误]
    end
    
    subgraph 处理方式
        Retry[自动重试]
        Notify[用户通知]
        Log[日志记录]
        Skip[跳过操作]
    end
    
    Auth --> Notify --> Log
    Network --> Retry --> Notify --> Log
    Data --> Log --> Skip
    Validation --> Log --> Notify
```

| 错误类型 | 错误码 | 处理策略 |
|----------|--------|----------|
| **Token 缺失** | `AUTH_TOKEN_MISSING` | 提示用户配置 |
| **Gist ID 无效** | `GIST_ID_MISSING` | 提示用户配置 |
| **网络超时** | `NETWORK_ERROR` | 重试 3 次 |
| **文件未找到** | `FILE_NOT_FOUND` | 提示检查 Gist |
| **数据格式错误** | `INVALID_DATA_FORMAT` | 尝试旧格式兼容 |
