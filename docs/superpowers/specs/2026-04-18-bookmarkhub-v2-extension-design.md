# BookmarkHub v2 扩展改造计划书

**日期:** 2026-04-18  
**状态:** 待审查  
**范围:** 浏览器扩展本体改造  
**目标参考:** [BookmarkHub 官网](https://bh.memoload.com/zh) 与 [定价页功能表](https://bh.memoload.com/zh/pricing#pricing_feature_table)

---

## 1. 项目目标

将当前的 BookmarkHub 从“以 Gist/WebDAV 同步为核心的工具型扩展”改造成“本地优先、可插拔多后端、具备安全与历史能力的产品型扩展”。

本次改造的目标是尽量高保真对齐目标产品在扩展侧体现出的核心能力：

- 多后端存储
- 端到端加密
- 智能多方合并
- 自动同步
- 同步历史
- 书签时光机
- 删除云端文件
- 配置云端同步
- 多浏览器一致体验

说明：以上是目标产品能力蓝图，不等同于首个可发布版本的必达范围。本计划按 `MVP`、`增强阶段` 与 `后续路线图` 分层推进，避免把长期愿景误写成一次性交付目标。

---

## 2. 已确认范围

### 2.1 本次纳入范围

- 仅改造浏览器扩展本体
- 按单一完整版本规划，不区分免费版/专业版
- 允许按新产品重新设计数据模型与配置体系
- 优先对齐目标产品的扩展能力，而不是兼容旧实现细节

### 2.2 本次明确不做

- 官网、下载页、文档站、社区页
- 登录/注册、账号体系、控制台
- 支付、订阅、许可校验
- 旧版 Gist/WebDAV 配置与数据的兼容迁移

### 2.3 设计原则

- 本地优先：书签读取、差异判断、加密与合并都在本地完成
- 统一抽象：所有远端后端都通过统一适配器接入
- 可追溯：每次同步、恢复、强制覆盖都要留痕
- 可恢复：时光机恢复不会抹掉历史，而是写入新的恢复记录
- 可扩展：新增存储后端时，不要求改动同步引擎和 UI 主流程

## 3. 当前项目现状评估

当前仓库已经具备以下基础：

- `WXT + React 18 + TypeScript` 的扩展工程结构
- `background.ts` 负责后台消息、同步触发、书签操作
- `popup` 和 `options` 两个基础界面
- `GitHub Gist` 与 `WebDAV` 两类存储后端
- 自动同步、导入导出、备份恢复、基础冲突处理
- 多语言资源目录

当前结构的主要限制：

- `background.ts` 与 `sync.ts` 仍偏工具型架构，职责较重
- 后端接入方式是“按类型写分支逻辑”，不适合扩展到多云后端
- 当前数据结构偏单次同步结果，不适合承载历史与时光机
- 自动同步、事件抑制、队列控制仍以运行时状态为中心，MV3 下扩展性有限
- UI 仍是“操作入口页”，不是“可观测、可恢复、可诊断”的产品控制面板

结论：现有项目适合作为能力来源与基础设施来源，但不适合继续沿当前同步内核直接叠加目标产品能力。建议在同仓库内按 v2 思路重建核心内核与 UI 结构。

## 4. 目标能力蓝图

### 4.1 首个可发布版本（MVP）

- 支持 `GitHub Repo / Gist / WebDAV`
- 支持手动同步
- 支持新增、删除、修改三类差异合并
- 支持基础同步历史查看
- 支持 Popup 状态展示与快速同步入口
- 支持 Options 中的存储配置、同步配置与错误反馈
- 完成 Chrome 与 Firefox 主链路验证

### 4.2 增强阶段目标

- 接入端到端加密
- 支持自动同步与强制同步
- 支持删除云端文件
- 增加基础同步诊断
- 为 `GitHub Repo / Gist` 提供完整时光机恢复

### 4.3 后续路线图（不纳入首发验收）

- 接入 `Gitee / GitLab / S3`
- 在认证模型稳定后评估 `Google Drive / OneDrive`
- 增加移动检测、冲突可视化和结果回放
- 逐步提升多浏览器一致体验

### 4.4 后端能力矩阵

- `GitHub Repo / Gist`
  - MVP: 手动同步、基础历史
  - 增强阶段: 加密、自动同步、强制同步、删除云端文件、完整时光机
- `WebDAV`
  - MVP: 手动同步、基础历史
  - 增强阶段: 加密、自动同步、强制同步、删除云端文件
  - 明确不承诺完整时光机
- `Gitee / GitLab / S3`
  - 作为后续非 OAuth 候选后端
- `Google Drive / OneDrive`
  - 作为后续 OAuth 候选后端，在认证链路完成前不进入实施范围

### 4.5 产品体验目标

- Popup 用于查看当前状态、最近结果与快捷操作
- Options 用于完整配置存储、安全、同步、历史与诊断
- 任何危险操作都必须有确认与反馈
- 同步结果需要可解释，而不只是“成功/失败”
- 当后端、加密或解密出现异常时，用户能知道失败在哪一步

## 5. 总体架构

建议采用五层架构，替代当前以 `background.ts + sync.ts` 为中心的集中式结构。

### 5.1 领域核心层 `domain`

职责：

- 定义统一数据模型
- 约束同步语义
- 描述冲突、历史、快照、恢复、加密封装

核心对象建议：

- `BookmarkSnapshot`
- `BookmarkNode`
- `SyncJournalEntry`
- `SyncSessionResult`
- `RemoteProfile`
- `ConflictRecord`
- `EncryptionEnvelope`
- `RestorePoint`

### 5.2 存储适配层 `adapters`

职责：

- 统一封装不同后端的数据面能力
- 对需要授权流程的后端补充控制面能力
- 向上提供稳定接口，不暴露后端差异

建议拆成两组接口：

#### 数据面接口 `StorageAdapter`

- `connect()`
- `healthcheck()`
- `readCurrentSnapshot()`
- `writeCurrentSnapshot()`
- `appendJournalEntry()`
- `listRestorePoints()`
- `readRestorePoint()`
- `deleteRemoteData()`

适用：

- `githubRepoAdapter`
- `githubGistAdapter`
- `webdavAdapter`
- 后续可扩展到 `giteeAdapter / gitlabAdapter / s3Adapter`

#### 认证控制面接口 `AuthCapableStorageAdapter`

- `beginAuthorization()`
- `handleAuthorizationCallback()`
- `refreshAuthorization()`
- `revokeAuthorization()`
- `getAuthorizationState()`

适用：

- `googleDriveAdapter`
- `oneDriveAdapter`

说明：在认证控制面设计与验证完成前，`Google Drive / OneDrive` 不应进入首批实施范围。

### 5.3 同步引擎层 `engine`

职责：

- 本地树标准化
- 差异检测
- 多方合并
- 冲突分类与决策
- 生成新快照与同步日志

关键要求：

- 不再只支持“上传/下载”二元模式
- MVP 先支持新增、删除、修改三类差异识别
- 移动检测、冲突可视化与结果回放属于后续里程碑
- MVP 先要求结果可记录、可解释，不要求完整回放

### 5.4 应用服务层 `app`

职责：

- 统一调度手动同步、自动同步、强制同步
- 管理后台队列、锁、事件抑制与错误恢复
- 维护 UI 状态广播
- 统一管理配置和加密状态

建议服务：

- `syncOrchestrator`
- `storageRegistry`
- `securityService`
- `historyService`
- `runtimeStateService`
- `notificationService`

### 5.5 界面层 `ui`

职责：

- 展示状态
- 收集配置
- 提供高频操作
- 呈现历史与诊断结果

界面建议：

- `popup`: 当前状态、最近同步结果、快速同步、异常提醒、跳转入口
- `options`: 存储配置、安全配置、同步策略、历史列表、时光机、远端删除、诊断信息

## 6. 模块拆分方案

建议将 v2 拆成 6 个高内聚模块。

### 6.1 Storage Center

负责：

- 后端配置管理
- 连接测试
- 远端快照读写
- 日志追加
- 恢复点查询
- 云端数据删除

### 6.2 Sync Engine

负责：

- 本地书签树标准化
- 三方或多方差异分析
- 冲突生成与处理
- 合并结果输出

### 6.3 Security Center

负责：

- 端到端加密
- 密钥派生
- 加密配置校验
- 解密错误提示

### 6.4 History Center

负责：

- 同步历史记录
- 时光机时间线
- 恢复预览
- 恢复执行

### 6.5 Runtime Orchestrator

负责：

- 手动同步
- 自动同步
- 浏览器事件调度
- 队列与锁
- MV3 唤醒后的状态恢复

### 6.6 Extension UI

负责：

- Popup 快速面板
- Options 控制台
- 状态展示与错误反馈

## 7. 数据模型设计方向

### 7.1 快照模型

`BookmarkSnapshot` 应包含：

- 统一后的书签树
- 版本号
- 生成时间
- 生成来源设备
- 哈希摘要
- 加密标记

### 7.2 日志模型

`SyncJournalEntry` 应包含：

- 会话 ID
- 时间戳
- 触发来源
- 本地摘要
- 远端摘要
- 合并策略
- 冲突摘要
- 结果摘要
- 是否为恢复动作

### 7.3 配置模型

`RemoteProfile` 应包含：

- 后端类型
- 标识名
- 所需参数
- 是否启用
- 是否启用历史
- 是否启用加密

### 7.4 加密模型

`EncryptionEnvelope` 应包含：

- 算法版本
- 盐值
- IV 或 nonce
- 密文
- 元数据校验信息

说明：需要区分“基础同步历史”和“完整时光机”。所有后端都可以保留基础同步历史，但完整时光机只应在具备足够快照或版本能力的后端开启。首个版本中，`WebDAV` 只承诺基础历史，不承诺完整时光机。

## 8. 关键数据流

### 8.1 首次配置

1. 用户选择后端类型
2. 输入并校验连接参数
3. 执行连接测试
4. 保存 `RemoteProfile`
5. 若启用加密，则初始化密钥材料
6. 生成本地运行时配置缓存

### 8.2 手动同步

1. Popup 发出同步请求
2. `Runtime Orchestrator` 拉取本地快照与远端快照
3. `Sync Engine` 执行差异分析与合并
4. `Security Center` 负责加密封装
5. `Storage Center` 写回当前快照与日志
6. `History Center` 更新历史视图
7. UI 收到广播并刷新状态

### 8.3 自动同步

1. 浏览器启动、书签事件或定时器触发
2. 统一进入调度入口
3. 检查锁、抑制状态、节流策略
4. 执行同步
5. 记录本次结果

### 8.4 时光机恢复（仅限支持时光机的后端）

1. 用户在 Options 查看时间线
2. 选择恢复点
3. 预览恢复目标
4. 确认后恢复为当前快照
5. 追加一条恢复日志
6. 刷新状态并提示结果

### 8.5 删除云端文件

1. 用户执行危险操作
2. 二次确认
3. 调用适配器删除云端数据
4. 保留本地审计记录
5. 更新 UI 状态

---

## 9. 错误处理与安全策略

### 9.1 错误处理原则

- 所有同步失败都必须带阶段信息
- 所有后端错误都要映射为统一错误类型
- 加密错误与连接错误必须区分
- 恢复失败不能覆盖现有快照

### 9.2 安全策略

- 所有敏感配置默认加密存储
- 同步前先本地加密，再上传远端
- 解密失败时不自动覆盖本地数据
- 危险操作需要明确确认
- 日志中禁止输出敏感凭据

### 9.3 MV3 运行约束

- 后台运行状态需要可恢复
- 自动同步不可依赖单次常驻内存状态
- 所有关键会话应可从持久化状态还原

## 10. 分期实施路线

### Phase 0: v2 基线重建

- 建立 `domain / adapters / engine / app / ui` 目录边界
- 定义核心模型与错误模型
- 建立测试基架与运行状态模型

阶段出口：

- 新目录边界建立完成
- `background` 到新编排层的最小调用链打通
- `npm run compile` 与 `npm run test` 继续可执行

### Phase 1: 最小可用 v2 内核

- 完成新的同步主链路
- 接入 `GitHub Repo / Gist / WebDAV`
- 重做 Popup 与 Options 骨架
- 支持基础历史查看
- 仅支持手动同步
- 合并范围仅覆盖新增、删除、修改

阶段出口：

- 三类后端都能完成手动同步
- Popup 可展示当前状态与最近一次结果
- Options 可配置三类后端并完成连接测试
- 基础历史可展示最近若干次同步记录
- 本阶段不包含自动同步、加密、完整时光机

### Phase 2: 安全与自动化

- 接入端到端加密
- 完成自动同步、强制同步、错误恢复
- 完善连接测试与安全状态展示
- 增加删除云端文件
- 为 `GitHub Repo / Gist` 增加完整时光机恢复

阶段出口：

- 三类后端可在启用加密后完成同步
- 自动同步在启动、事件和定时触发下行为一致
- 删除云端文件具备二次确认与明确反馈
- `GitHub Repo / Gist` 完成恢复预览与时光机恢复
- `WebDAV` 明确保持“基础历史，无完整时光机”

### Phase 3: 非 OAuth 后端扩展

- 接入 `Gitee / GitLab / S3`
- 校验各后端在统一接口下的一致性

阶段出口：

- 新后端全部通过共享适配器一致性测试
- UI 不因新增后端而产生专门分支页面
- 新增后端不影响 MVP 与增强阶段已有行为

### Phase 4: OAuth 后端可行性与接入

- 完成认证控制面设计
- 评估并接入 `Google Drive / OneDrive`
- 验证令牌刷新、撤销、权限探测与异常恢复

阶段出口：

- OAuth 接口设计定稿
- 至少一个 OAuth 后端完成从授权到同步的闭环验证
- 若闭环不稳定，则保留为研究分支，不并入主线

### Phase 5: 高级同步能力封装

- 增加移动检测
- 增加冲突可视化
- 增加结果回放
- 增强同步诊断

阶段出口：

- 移动检测在主要浏览器上有明确行为定义
- 冲突结果可被用户理解和处理
- 回放能力不影响现有同步性能

### Phase 6: 发布级收尾

- 测试矩阵补齐
- 构建、性能与并发验证
- 国际化补全
- 发布文档与离线安装验证

## 12. 风险与应对

### 风险 1：多后端能力不一致

应对：

- 统一能力矩阵
- 将“原生支持”和“扩展模拟支持”明确分层

### 风险 2：历史与时光机在部分后端上碎片化

应对：

- 所有后端统一保留基础同步历史
- 完整时光机只在能力足够的后端开启
- `WebDAV` 在计划层面明确排除完整时光机承诺

### 风险 3：MV3 下自动同步不稳定

应对：

- 重构后台状态管理
- 将会话状态持久化

### 风险 4：加密引入使用门槛

应对：

- 提供清晰的启用流程与恢复提示
- 明确忘记密码后的边界

### 风险 5：继续沿旧结构堆功能导致复杂度失控

应对：

- 先完成 v2 基线重建
- 以模块边界作为硬约束

## 13. 验收标准

### 13.1 MVP 验收标准

- 扩展支持 `GitHub Repo / Gist / WebDAV`
- 三类后端都能完成手动同步
- 合并范围覆盖新增、删除、修改
- Popup 能展示当前状态与最近一次结果
- Options 能完成三类后端配置与连接测试
- 基础历史可展示最近若干次同步记录
- Chrome 与 Firefox 主链路验证通过
- `npm run compile`、`npm run test`、`npm run build` 可通过

### 13.2 增强阶段验收标准

- 用户可以启用端到端加密，并完成至少一次加密同步与解密恢复
- 手动同步、自动同步、强制同步行为一致且可观测
- 删除云端文件具备二次确认、明确反馈与审计记录
- `GitHub Repo / Gist` 支持完整时光机恢复
- `WebDAV` 明确仅支持基础历史，不支持完整时光机
- 关键模块具备单测与集成验证

### 13.3 不纳入当前验收

- `Gitee / GitLab / S3 / Google Drive / OneDrive`
- 移动检测
- 冲突可视化
- 结果回放

## 14. 实施建议

推荐路线：在同一仓库内按 v2 思路重建核心内核与页面结构，只复用当前仓库中仍健康的基础能力，如书签读取、多语言资源、构建配置与部分工具函数。

不建议直接在当前 `background.ts` 与 `sync.ts` 之上继续堆叠目标产品能力，否则短期可用，长期维护成本会迅速上升。
