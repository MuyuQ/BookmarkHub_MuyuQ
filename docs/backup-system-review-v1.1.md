## 二、中等问题（建议修复）

### 2.1 上传流程步骤3-4的顺序问题

**问题描述**：

上传流程中：

```
│ 2. 获取浏览器当前所有书签                          │
│    currentBrowserBookmarks = getBrowserBookmarks()│
│                                                  │
│ 3. 获取远程数据                                   │
│    remoteData = await fetchFromRemoteGist()       │
```

**问题**：
- 先获取书签，再获取远程数据
- 如果远程数据获取失败，用户可能在这期间又修改了书签
- 下次重试时，书签数据已经不是用户期望的版本

**建议**：

考虑在远程数据获取成功后，再次获取书签（或接受当前设计，因为这是小概率事件）：

```
│ 3. 获取远程数据                                   │
│    remoteData = await fetchFromRemoteGist()       │
│    if (获取失败) 结束流程                         │
│                                                  │
│ 4. 获取浏览器当前所有书签                          │
│    currentBrowserBookmarks = getBrowserBookmarks()│
```

**优先级**: P2（可选优化）

### 2.2 防抖机制中的计时器持久化问题

**问题描述**：

防抖代码中：

```typescript
class SyncDebouncer {
  private timer: ReturnType<typeof setTimeout> | null = null;
  private maxTimer: ReturnType<typeof setTimeout> | null = null;
  private pendingSync: boolean = false;
```

**问题**：
- Service Worker 休眠后，计时器会被清除
- 但 `pendingSync` 标志也会丢失
- 可能导致待执行的同步被遗忘

**影响**：
- 小概率事件，用户快速操作后扩展被休眠
- 下次唤醒后，pendingSync 丢失

**建议**：

将 `pendingSync` 也持久化到 `browser.storage.local`：

```typescript
// 在 executeSync 中
if (this.pendingSync) {
  await browser.storage.local.set({ pendingSync: true });
}

// 在 triggerSync 开始时检查
const storedPendingSync = await browser.storage.local.get('pendingSync');
if (storedPendingSync.pendingSync) {
  // 立即触发一次同步
}
```

**优先级**: P2（可选优化，影响小概率边界情况）

**建议**：

在文档中补充跨设备同步最佳实践说明：

3. 下载前会自动备份当前书签到本地历史

## 四、设计亮点

### 4.1 数据一致性保障

v1.1 的数据流顺序设计非常合理：

| 操作 | 顺序 | 优势 |
|-----|------|------|
| 上传 | 远程操作 → 本地缓存不变 | 避免本地缓存被意外覆盖 |
| 下载 | 替换书签 → 保存缓存 | 避免书签替换失败但缓存已保存 |
| 还原 | 替换书签 → 保存缓存 → 远程同步 | 本地优先，网络失败不影响本地 |

### 4.2 锁机制设计

- 使用 `browser.storage.local` 持久化锁
- 锁超时自动清理，防止死锁
- 锁重入保护，防止并发问题

### 4.3 防抖机制设计

- 双计时器机制（debounceTime + maxWaitTime）
- 既避免频繁同步，又保证最终一致性
- pendingSync 机制确保不丢失用户操作

### 4.4 错误处理设计

- 分级错误提示
- 网络失败不影响本地数据
- 用户友好的错误消息
