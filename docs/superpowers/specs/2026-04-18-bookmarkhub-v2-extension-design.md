# BookmarkHub v2 扩展改造计划书

**日期:** 2026-04-18  
**状态:** 待审查  
**范围:** 浏览器扩展本体改造  
**目标参考:** [BookmarkHub 官网](https://bh.memoload.com/zh) 与 [定价页功能表](https://bh.memoload.com/zh/pricing#pricing_feature_table)

---

## 1. 项目目标

将当前的 BookmarkHub 从“以 Gist/WebDAV 同步为核心的工具型扩展”改造成“本地优先、可插拔多后端、具备安全与历史能力的产品型扩展”。

本次改造的目标是尽量高保真对齐目标产品在扩展侧体现出的核心能力：

- 多后端存储
- 端到端加密
- 智能多方合并
- 自动同步
- 同步历史
- 书签时光机
- 删除云端文件
- 配置云端同步
- 多浏览器一致体验

---

## 2. 已确认范围

### 2.1 本次纳入范围

- 仅改造浏览器扩展本体
- 按单一完整版本规划，不区分免费版/专业版
- 允许按新产品重新设计数据模型与配置体系
- 优先对齐目标产品的扩展能力，而不是兼容旧实现细节

### 2.2 本次明确不做

- 官网、下载页、文档站、社区页
- 登录/注册、账号体系、控制台
- 支付、订阅、许可校验
- 旧版 Gist/WebDAV 配置与数据的兼容迁移

### 2.3 设计原则

- 本地优先：书签读取、差异判断、加密与合并都在本地完成
- 统一抽象：所有远端后端都通过统一适配器接入
- 可追溯：每次同步、恢复、强制覆盖都要留痕
- 可恢复：时光机恢复不会抹掉历史，而是写入新的恢复记录
- 可扩展：新增存储后端时，不要求改动同步引擎和 UI 主流程

---

## 3. 当前项目现状评估

当前仓库已经具备以下基础：

- `WXT + React 18 + TypeScript` 的扩展工程结构
- `background.ts` 负责后台消息、同步触发、书签操作
- `popup` 和 `options` 两个基础界面
- `GitHub Gist` 与 `WebDAV` 两类存储后端
- 自动同步、导入导出、备份恢复、基础冲突处理
- 多语言资源目录

当前结构的主要限制：

- `background.ts` 与 `sync.ts` 仍偏工具型架构，职责较重
- 后端接入方式是“按类型写分支逻辑”，不适合扩展到多云后端
- 当前数据结构偏单次同步结果，不适合承载历史与时光机
- 自动同步、事件抑制、队列控制仍以运行时状态为中心，MV3 下扩展性有限
- UI 仍是“操作入口页”，不是“可观测、可恢复、可诊断”的产品控制面板

结论：现有项目适合作为能力来源与基础设施来源，但不适合继续沿当前同步内核直接叠加目标产品能力。建议在同仓库内按 v2 思路重建核心内核与 UI 结构。

## 4. 目标能力蓝图

### 4.1 必达能力

- 支持 `GitHub Repo / Gist / Gitee / GitLab / Google Drive / OneDrive / S3 / WebDAV`
- 支持手动同步、自动同步、强制同步
- 支持端到端加密
- 支持同步历史查看
- 支持书签时光机恢复
- 支持删除云端文件
- 支持配置导入、导出与远端同步策略设置
- 支持 Chrome / Firefox / Edge / Chromium 系列浏览器

### 4.2 产品体验目标

- Popup 用于查看当前状态、最近结果与快捷操作
- Options 用于完整配置存储、安全、同步、历史与诊断
- 任何危险操作都必须有确认与反馈
- 同步结果需要可解释，而不只是“成功/失败”
- 当后端、加密或解密出现异常时，用户能知道失败在哪一步

## 5. 总体架构

建议采用五层架构，替代当前以 `background.ts + sync.ts` 为中心的集中式结构。

### 5.1 领域核心层 `domain`

职责：

- 定义统一数据模型
- 约束同步语义
- 描述冲突、历史、快照、恢复、加密封装

核心对象建议：

- `BookmarkSnapshot`
- `BookmarkNode`
- `SyncJournalEntry`
- `SyncSessionResult`
- `RemoteProfile`
- `ConflictRecord`
- `EncryptionEnvelope`
- `RestorePoint`

### 5.2 存储适配层 `adapters`

职责：

- 统一封装不同后端的连接、读写、列举、删除、历史能力
- 向上提供稳定接口，不暴露后端差异

建议接口：

- `connect()`
- `healthcheck()`
- `readCurrentSnapshot()`
- `writeCurrentSnapshot()`
- `appendJournalEntry()`
- `listRestorePoints()`
- `readRestorePoint()`
- `deleteRemoteData()`

适配器：

- `githubRepoAdapter`
- `githubGistAdapter`
- `giteeAdapter`
- `gitlabAdapter`
- `googleDriveAdapter`
- `oneDriveAdapter`
- `s3Adapter`
- `webdavAdapter`

### 5.3 同步引擎层 `engine`

职责：

- 本地树标准化
- 差异检测
- 多方合并
- 冲突分类与决策
- 生成新快照与同步日志

关键要求：

- 不再只支持“上传/下载”二元模式
- 必须能识别新增、删除、修改、移动
- 合并结果必须可回放、可记录

### 5.4 应用服务层 `app`

职责：

- 统一调度手动同步、自动同步、强制同步
- 管理后台队列、锁、事件抑制与错误恢复
- 维护 UI 状态广播
- 统一管理配置和加密状态

建议服务：

- `syncOrchestrator`
- `storageRegistry`
- `securityService`
- `historyService`
- `runtimeStateService`
- `notificationService`

### 5.5 界面层 `ui`

职责：

- 展示状态
- 收集配置
- 提供高频操作
- 呈现历史与诊断结果

界面建议：

- `popup`: 当前状态、最近同步结果、快速同步、异常提醒、跳转入口
- `options`: 存储配置、安全配置、同步策略、历史列表、时光机、远端删除、诊断信息

## 6. 模块拆分方案

建议将 v2 拆成 6 个高内聚模块。

### 6.1 Storage Center

负责：

- 后端配置管理
- 连接测试
- 远端快照读写
- 日志追加
- 恢复点查询
- 云端数据删除

### 6.2 Sync Engine

负责：

- 本地书签树标准化
- 三方或多方差异分析
- 冲突生成与处理
- 合并结果输出

### 6.3 Security Center

负责：

- 端到端加密
- 密钥派生
- 加密配置校验
- 解密错误提示

### 6.4 History Center

负责：

- 同步历史记录
- 时光机时间线
- 恢复预览
- 恢复执行

### 6.5 Runtime Orchestrator

负责：

- 手动同步
- 自动同步
- 浏览器事件调度
- 队列与锁
- MV3 唤醒后的状态恢复

### 6.6 Extension UI

负责：

- Popup 快速面板
- Options 控制台
- 状态展示与错误反馈

## 7. 数据模型设计方向

### 7.1 快照模型

`BookmarkSnapshot` 应包含：

- 统一后的书签树
- 版本号
- 生成时间
- 生成来源设备
- 哈希摘要
- 加密标记

### 7.2 日志模型

`SyncJournalEntry` 应包含：

- 会话 ID
- 时间戳
- 触发来源
- 本地摘要
- 远端摘要
- 合并策略
- 冲突摘要
- 结果摘要
- 是否为恢复动作

### 7.3 配置模型

`RemoteProfile` 应包含：

- 后端类型
- 标识名
- 所需参数
- 是否启用
- 是否启用历史
- 是否启用加密

### 7.4 加密模型

`EncryptionEnvelope` 应包含：

- 算法版本
- 盐值
- IV 或 nonce
- 密文
- 元数据校验信息

说明：历史与时光机不应完全依赖后端原生版本能力。即便后端不支持版本控制，扩展也应通过“当前快照 + 追加式同步日志”的方式提供统一的历史能力。

## 8. 关键数据流

### 8.1 首次配置

1. 用户选择后端类型
2. 输入并校验连接参数
3. 执行连接测试
4. 保存 `RemoteProfile`
5. 若启用加密，则初始化密钥材料
6. 生成本地运行时配置缓存

### 8.2 手动同步

1. Popup 发出同步请求
2. `Runtime Orchestrator` 拉取本地快照与远端快照
3. `Sync Engine` 执行差异分析与合并
4. `Security Center` 负责加密封装
5. `Storage Center` 写回当前快照与日志
6. `History Center` 更新历史视图
7. UI 收到广播并刷新状态

### 8.3 自动同步

1. 浏览器启动、书签事件或定时器触发
2. 统一进入调度入口
3. 检查锁、抑制状态、节流策略
4. 执行同步
5. 记录本次结果

### 8.4 时光机恢复

1. 用户在 Options 查看时间线
2. 选择恢复点
3. 预览恢复目标
4. 确认后恢复为当前快照
5. 追加一条恢复日志
6. 刷新状态并提示结果

### 8.5 删除云端文件

1. 用户执行危险操作
2. 二次确认
3. 调用适配器删除云端数据
4. 保留本地审计记录
5. 更新 UI 状态

---

## 9. 错误处理与安全策略

### 9.1 错误处理原则

- 所有同步失败都必须带阶段信息
- 所有后端错误都要映射为统一错误类型
- 加密错误与连接错误必须区分
- 恢复失败不能覆盖现有快照

### 9.2 安全策略

- 所有敏感配置默认加密存储
- 同步前先本地加密，再上传远端
- 解密失败时不自动覆盖本地数据
- 危险操作需要明确确认
- 日志中禁止输出敏感凭据

### 9.3 MV3 运行约束

- 后台运行状态需要可恢复
- 自动同步不可依赖单次常驻内存状态
- 所有关键会话应可从持久化状态还原

## 12. 风险与应对

### 风险 1：多后端能力不一致

应对：

- 统一能力矩阵
- 将“原生支持”和“扩展模拟支持”明确分层

### 风险 2：历史与时光机在部分后端上碎片化

应对：

- 所有后端统一采用“当前快照 + 追加日志”模型
- 原生版本控制仅作为增强能力

### 风险 3：MV3 下自动同步不稳定

应对：

- 重构后台状态管理
- 将会话状态持久化

### 风险 4：加密引入使用门槛

应对：

- 提供清晰的启用流程与恢复提示
- 明确忘记密码后的边界

### 风险 5：继续沿旧结构堆功能导致复杂度失控

应对：

- 先完成 v2 基线重建
- 以模块边界作为硬约束

## 13. 验收标准

- 扩展支持统一配置并稳定运行 `GitHub Repo / Gist / Gitee / GitLab / Google Drive / OneDrive / S3 / WebDAV`
- 用户可以启用端到端加密，并完成至少一次加密同步与解密恢复
- 用户可以查看同步历史与时间线
- 用户可以从历史点恢复书签，并生成新的恢复记录
- 手动同步、自动同步、强制同步行为一致且可观测
- Popup 能展示当前状态、最近结果与异常
- Options 能完成存储、安全、同步、历史与清理相关配置
- Chrome 与 Firefox 主链路验证通过
- 关键模块具备单测与集成验证

---

## 14. 实施建议

推荐路线：在同一仓库内按 v2 思路重建核心内核与页面结构，只复用当前仓库中仍健康的基础能力，如书签读取、多语言资源、构建配置与部分工具函数。

不建议直接在当前 `background.ts` 与 `sync.ts` 之上继续堆叠目标产品能力，否则短期可用，长期维护成本会迅速上升。
